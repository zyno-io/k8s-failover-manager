# k8s-failover-manager - AI Agent Guide

## Project Overview

Kubernetes operator (Kubebuilder 4.x / controller-runtime) that orchestrates DNS-based failover across two clusters. A `FailoverService` CRD controls a managed Service, executing transition actions (HTTP, Job, Scale, WaitReady) through a state machine, and killing stale connections via a DaemonSet agent.

**API Group:** `k8s-failover.zyno.io` | **Version:** `v1alpha1` | **Kind:** `FailoverService`

## Key Architecture Concepts

### Transition State Machine

The controller uses a phase-based state machine in `status.transition.phase`:

```
"" (Idle) ‚Üí ExecutingPreActions ‚Üí UpdatingResources ‚Üí FlushingConnections ‚Üí ExecutingPostActions ‚Üí "" (Idle)
```

- **Idle:** Watches for `spec.failoverActive` changes; runs anti-entropy sync every 60s
- **ExecutingPreActions:** Runs pre-actions sequentially; requeues after 5s for async actions (Jobs)
- **UpdatingResources:** Updates the managed Service to point to the new target address
- **FlushingConnections:** Signals connection-killer DaemonSet via ConfigMap
- **ExecutingPostActions:** Runs post-actions sequentially; completes transition on finish

### Address Resolution Logic

```
if clusterRole == "primary" ‚Üí use spec.primaryCluster
if clusterRole == "failover" ‚Üí use spec.failoverCluster

if spec.failoverActive == false ‚Üí use target.primaryModeAddress
if spec.failoverActive == true  ‚Üí use target.failoverModeAddress
```

- DNS addresses ‚Üí ExternalName Service
- IP addresses ‚Üí ClusterIP Service + managed EndpointSlice

### Transition Hook Resolution

Each cluster has its own hooks (`onFailover` / `onFailback`). The controller selects hooks based on its own `CLUSTER_ROLE`, not the remote cluster's. Both clusters run transitions independently with their own action sets.

### Multi-Cluster Sync

- Push-based: after completing a transition, the controller updates the remote cluster's FailoverService spec
- Conflict resolution: higher `status.activeGeneration` wins; on tie, lexicographic cluster ID comparison
- Remote kubeconfig is read from a Kubernetes Secret, cached with resource-version-based invalidation

### Connection Killing

- Controller writes kill signals to a ConfigMap (`connection-kill-signal`) keyed by node name
- DaemonSet agent watches the ConfigMap, uses netlink to destroy TCP/UDP connections to old IPs
- Agent acknowledges completion via the same ConfigMap
